# MCode 本地索引与 Git SCM 痛点及优化方案

本文档梳理了当前 MCode 本地索引（RAG）以及 Git SCM 集成中存在的五个核心痛点，并提供了相应的技术优化方案。通过解决这些痛点，可以显著提升系统的检索精度、检索性能、运行时稳定性以及用户数据安全性。

---

## 1. 🔍 HNSW 过滤稀疏性回退机制 (HNSW Filter Sparsity Fallback)

### 痛点描述
当启用 RAG 意图路由（Router）过滤特定的文档类型（`docType`，例如仅查询 `git_commit` 或 `doc_chunk`）时，系统会从 HNSW 向量索引中检索前 $K$ 个全局近邻，然后通过 `docTypes!.includes(record.docType)` 进行**后过滤**（Post-filtering）。
* **缺陷**：如果这 $K$ 个全局近邻的类型全部是 `code_chunk`（这在大型代码库中非常普遍），那么过滤后将没有任何结果保留（返回 0 个结果），**即使数据库中明明存在与查询相关的其它类型记录**。这就造成了检索的“稀疏空洞”。

### 优化方案
引入**自动回退暴力扫描**的双通道机制：
1. 在 HNSW 检索并过滤后，如果返回的结果数量少于期望的 `options.topK`，系统应自动执行降级逻辑。
2. 降级逻辑自动回退到暴力扫描 `similaritySearchBruteForce`，因为暴力扫描是在 SQL 层面进行预过滤（`WHERE doc_type IN (...)`），能百分之百确保召回当前类型下最相关的 $K$ 个向量，消灭检索空洞。

## 2. ⚡ SQLite 暴力检索的延迟加载优化 (Deferred Row Loading)

### 痛点描述
如果 HNSW 索引不可用或触发回退，系统会执行暴力检索。当前的扫描查询为：
```sql
SELECT chunk_id, file_path, doc_type, text_content, metadata_json, embedding, dim, hnsw_key FROM rag_chunks ...
```
* **缺陷**：在每一轮批量扫描（Batch Size = 512）中，它会把所有分片的 `text_content` 和 `metadata_json`（大字符串）加载并解析到 V8 内存中。在包含数万个 chunk 的大项目里，这会导致巨大的磁盘 I/O 吞吐、频繁的 GC 内存抖动以及高昂的字符串解码 CPU 开销。

### 优化方案
采用**两阶段延迟加载（Deferred Loading）**策略：
1. **第一阶段（计算相似度）**：在 SQL 查询中仅载入用于计算的轻量级字段：
   ```sql
   SELECT chunk_id, doc_type, embedding, dim FROM rag_chunks
   ```
   在 JS 内存中对这些轻量级数据快速遍历、计算点积，并使用 Heap 筛选出排名前 $K$ 的 `chunk_id`。
2. **第二阶段（读取完整内容）**：通过一次精准的批量查询加载最终优胜者的完整字段：
   ```sql
   SELECT * FROM rag_chunks WHERE chunk_id IN (...)
   ```
   这能降低 90% 以上的 I/O 和内存开销，提升暴力扫描速度 **10x ~ 50x**。

## 3. 🛡️ 规避 Float32Array 内存未对齐导致的 RangeError 崩溃

### 痛点描述
在将 SQLite 存储的二进制 Buffer 还原为向量数组时，代码中使用了：
```typescript
return new Float32Array(buffer.buffer, buffer.byteOffset, dim);
```
* **缺陷**：在 Node.js 中，Buffer 通常是从一个共享的内存池（Buffer Pool）中 slice 分配出来的。这意味着 `buffer.byteOffset` **并不一定能整除 4**（非 4 字节对齐）。如果在字节偏移未对齐的 ArrayBuffer 上直接实例化 `Float32Array`，JavaScript 会抛出致命的 `RangeError: start offset of Float32Array must be a multiple of 4`，导致索引加载或检索直接崩溃。

### 优化方案
在创建 TypedArray 视图前进行对齐检测，如果未对齐则复制内存：
```typescript
export function embeddingBufferToFloat32(buffer: Buffer, dim: number): Float32Array {
    if (buffer.byteLength !== dim * 4) {
        throw new Error(`[RAG] Embedding buffer size mismatch: expected ${dim * 4}, got ${buffer.byteLength}`);
    }
    // 4字节对齐校验
    if (buffer.byteOffset % 4 !== 0) {
        // slice 会在内存中复制一份数据，生成一个从 0 开始且对齐的独立 ArrayBuffer
        const alignedBuffer = buffer.buffer.slice(buffer.byteOffset, buffer.byteOffset + buffer.byteLength);
        return new Float32Array(alignedBuffer, 0, dim);
    }
    return new Float32Array(buffer.buffer, buffer.byteOffset, dim);
}
```

## 4. 📂 `.mcodeignore` 忽略规则的 Glob 匹配合规化

### 痛点描述
当前在 `isPathIgnored` 中，忽略判断使用了简单的字符串包含和特定结尾字符判断：
```typescript
relativePath === cleanPattern ||
relativePath.startsWith(cleanPattern + '/') ||
(cleanPattern.endsWith('*') && relativePath.startsWith(cleanPattern.slice(0, -1)))
```
* **缺陷**：它无法解析标准的 Glob 语法。如果用户在忽略文件中写入了常见规则（如 `*.log`、`*.tmp`、`**/build`、`dist/**/*.js`），当前匹配器完全无法识别，会导致临时文件或大型生成产物被错误索引。

### 优化方案
引入轻量级的 `picomatch` / `minimatch` 依赖，或在本地实现符合 `.gitignore` 规范的简易通配符-正则转换函数，确保通配符配置能正确过滤无效文件。

## 5. 🔄 Git SCM 撤销操作的安全防护 (SCM Undo Safety)

### 痛点描述
目前，MCode 提供的自动 Commit 每次使用 `git add -A` 暂存所有改动，而一键撤销（Undo）则是通过硬重置命令：
```typescript
await git('git reset --hard HEAD~1', path);
```
* **缺陷**：`git reset --hard` 会把整个工作区重置。如果用户在让 AI 修改代码前，工作区里本身就存有**自己未 commit 的手写修改**，一旦用户点击“Undo”，这些手写修改会被彻底抹除且**无法恢复**，面临极高的数据丢失风险。

### 优化方案
1. **精准暂存**：AI 写入文件时，仅对**被 AI 修改的文件**执行 `git add <filePath>`，而不是 `git add -A` 暂存整个工作区。
2. **非破坏性回滚**：在执行撤销操作时，放弃 `git reset --hard`，改用只针对 AI 修改文件的单文件签出或回滚：
   ```typescript
   await git(`git checkout HEAD~1 -- ${modifiedFiles.join(' ')}`, path);
   ```
   这能保证只撤销 AI 的本次操作，而用户未提交的其它手写文件可以安全保留。

## 6. ⚡ 代码扫描顺序优化 (Prioritized Directory Scanning)

### 痛点描述
在大型项目中，代码扫描和构建索引通常耗时较长。当前在 `buildLocalIndex` 和 `collectTargetFilesAsync` 中，扫描返回的文件列表顺序取决于磁盘遍历的自然顺序。
* **缺陷**：核心业务逻辑目录（如 `src/`）与边缘辅助目录（如 `docs/`、`tests/`、`scripts/` 等）混合在一起。如果扫描在中途被中断或处于冷启动阶段，最需要被 RAG 引用的核心业务代码可能需要等待整个扫描结束才能被检索到，极大地降低了 RAG 索引的冷启动可用性。

### 优化方案
引入**核心目录优先级排序（Directory Prioritization）**：
1. 在收集到所有目标文件并筛选出 `pendingFiles` 后，对其进行重新排序：
   - 提取相对路径中属于核心目录（如 `src/`）的文件，将其排在队列的最前端。
   - 其余非核心目录（如文档、工具链、测试代码等）靠后排列。
2. 这样可以确保系统在扫描开始的黄金几分钟内，优先将最核心的业务代码向量化，让用户能以最快速度在主要模块上用起 RAG 功能。

## 7. 🧪 测试目录及测试文件黑名单优化 (Test Directories Blacklisting)

### 痛点描述
在绝大多数软件项目中，`test` 或 `tests` 目录下存放的都是单元测试、集成测试或测试数据文件。
* **缺陷**：这些测试代码不包含核心业务逻辑。如果在扫描工作区构建索引时将它们扫入向量库，一方面会产生大量冗余的分片和 Embedding 调用开销，另一方面在回答业务逻辑问题时，测试代码容易作为相似噪音被召回，降低 RAG 回答的准确度。

### 优化方案
引入**测试目录自动忽略（Test Exclude）**机制：
1. 将 `test` 和 `tests` 目录显式加入静态跳过列表 `SKIPPED_DIRS` 中。
2. 这样，不论是在目录深度遍历扫描阶段，还是在单文件增量更新检查阶段，任何包含 `test` 或 `tests` 路径段的文件夹和文件都将被自动过滤，不参与向量索引的构建，保持索引的纯净度并节省算力。
