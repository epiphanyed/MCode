# 设计文档：任务拆分交互优化

## 1. 背景

Agent 分析大型任务时常先输出「任务分解」计划，例如：

```
任务分解：
创建 svg.md 文档框架
分析 SVG 解析器核心类 - CSvgParser, CSvgFile, SvgReader
…
```

当前问题：

| 问题 | 说明 |
|------|------|
| 纯文本列表 | 无法勾选、无完成进度 |
| Markdown `- [ ]` 只读 | `ChatMarkdownRender` 渲染 checkbox 但 `readOnly` |
| 无 Cursor 式任务卡 | 缺少折叠、进度、运行中高亮 |

用户期望：**类似 Cursor To-do 列表**——可交互勾选、可见完成度、Agent 运行时可对照进度。

## 2. 目标

- 自动识别 Assistant 消息中的**任务分解块**（中文/英文标题 + 列表）。
- 渲染为 **TaskPlanPanel**：可折叠、可勾选、显示 `已完成/总数`。
- 勾选状态**持久化到 thread**，刷新/切换线程后保留。
- 支持 Markdown 任务语法 `- [ ]` / `- [x]` 的**可交互** checkbox。
- Agent 运行中在输入框上方显示**当前任务计划**（sticky 摘要条）。
- Prompt 引导 LLM 用标准任务列表格式输出。

## 3. 任务块识别规则

### 3.1 带标题的任务分解（主路径）

匹配行（独立一行）：

- `任务分解` / `任务拆解` / `任务计划`
- `Task Plan` / `Task Breakdown` / `TODO` / `Plan`
- 可选 `#` 标题前缀、可选中英文冒号

标题之后连续行视为任务项，直到：

- 遇到新的 Markdown 标题（`# …`）
- 连续两个空行
- 出现明显非列表段落（>80 字且无列表前缀）

任务项行格式（任一）：

1. `- [ ] 描述` / `- [x] 描述`（GFM task list）
2. `1. 描述` / `- 描述` / `• 描述`
3. **纯文本行**（标题模式下，≥3 字符的非空行）

至少 **2 条**任务项才渲染为 TaskPlanPanel，避免误判。

### 3.2 无标题的 GFM 任务列表

消息中连续 ≥3 条 `- [ ]` / `- [x]` 且无任务卡时，标题默认为「Tasks」。

## 4. 数据模型

```typescript
type TaskPlanItem = {
  id: string          // `${index}-${hash(text)}` 稳定 id
  text: string
  initialChecked: boolean  // 来自 LLM 的 [x]
}

type TaskPlan = {
  title: string
  items: TaskPlanItem[]
}
```

Thread 状态（持久化）：

```typescript
taskPlanChecked: Record<string, boolean>
// key: `${messageIdx}:${itemId}`

taskPlanCollapsed: Record<string, boolean>
// key: `${messageIdx}` → 折叠状态
```

## 5. UI 设计（Cursor 风格）

```
┌─ 任务分解 ─────────────────── 2/5 ─┐
│ ▼                                   │
│ ○ 创建 svg.md 文档框架              │
│ ● 分析 CSvgParser …    (done)       │
│ ○ 分析 SvgObjects …                 │
└─────────────────────────────────────┘
```

- 头部：Chevron + 标题 + 进度徽章；点击折叠/展开。
- 行：圆形 checkbox（未完成）/ 勾选（完成）；完成项删除线 + 降透明度。
- Agent 运行中：sticky 条显示同计划 + 进度（取自最新含任务分解的 assistant 消息）。

## 6. 服务 API（IChatThreadService）

```typescript
getTaskPlanItemChecked(threadId, messageIdx, itemId, defaultChecked): boolean
setTaskPlanItemChecked(threadId, messageIdx, itemId, checked): void
toggleTaskPlanItem(threadId, messageIdx, itemId, defaultChecked): void
isTaskPlanCollapsed(threadId, messageIdx): boolean
setTaskPlanCollapsed(threadId, messageIdx, collapsed): void
```

解析逻辑放在 `common/helpers/taskPlanParser.ts`（纯函数，UI/服务共用）。

## 7. 渲染集成

```
AssistantMessageComponent
  ├─ extractTaskPlanFromMarkdown(displayContent)
  ├─ TaskPlanPanel (if plan)
  └─ ChatMarkdownRender(bodyWithoutPlan)
```

`ChatMarkdownRender` 中 GFM list `item.task` 改为可点击，回调 `toggleTaskPlanItem`（itemId 用 list index + text hash）。

## 8. Agent Prompt

在 `agent` 模式 system/details 增加：

> 复杂任务开始前输出任务分解，使用 `- [ ]` 列表，每完成一步可将对应项改为 `- [x]`（用户也可在 UI 勾选）。

## 9. 非目标

- 不做独立 `update_task` 工具（v1 仅 UI + 解析）。
- 不同步到外部 issue tracker。
- 不修改 LLM 上下文中的勾选状态（仅 UI 状态；后续可选注入 prompt）。

## 11. 测试要点

1. 用户示例（「任务分解：」+ 纯文本行）→ 任务卡 5 项。
2. 勾选持久化，切换线程再回来状态保留。
3. `- [ ]` markdown 列表可点击。
4. 少于 2 项不误触发任务卡。
5. 流式输出过程中任务项逐渐增加，已勾选项不丢失（id 按 text hash）。
