# 设计方案：Agent 修改文件增强（延迟批量 Review）

## 1. 背景与痛点

写分析文档（如 `代码分析/*.md`）时，Agent 往往需要：

1. 多次 `edit_file` / `rewrite_file` 增量写入同一 deliverable；
2. 每轮 edit 之间穿插 `read_file` / `search_for_files` 收集信息。

现有流程存在两层「审核」：

| 环节 | 触发条件 | 表现 |
|------|----------|------|
| **工具权限审核** | `autoApprove.edits === false` | Agent 在每次 `edit_file` 前暂停，`tool_request` + Approve/Cancel |
| **Diff 接受审核** | `autoAcceptLLMChanges === false` | 每次 edit 完成后，侧边栏该 tool 卡片出现 Keep/Remove；编辑器内保留 diff 高亮 |

用户期望：**Agent 运行期间不中断**，同一文件可多次修改；**任务结束后再统一 Review**（Accept All / Reject All）。

## 2. 目标

- Agent 模式下，`edit_file` / `rewrite_file` **不再逐次弹出工具权限审核**（可配置）。
- Agent 运行中 **隐藏** 每条 edit tool 的 Keep/Remove，避免干扰。
- Agent 正常结束（LLM 不再调用 tool）后，若仍有 pending diff → 进入 **`awaiting_batch_review`**，展示批量审核条。
- **同一文件多次 edit**：沿用 `editCodeService` 的 `keep-conflicts` 链式 apply，baseline 保持为首次 edit 前快照，文件内容为累积结果。
- 与现有 **命令栏 Accept/Reject All** 兼容，批量条与之联动。

## 3. 非目标

- 不改变 `normal` / `gather` 模式的审核行为。
- 不替代 `autoAcceptLLMChanges`（用户若开启仍会在每次 edit 后立即 accept）。
- 不改动 `create_file_or_folder` 等其它 edits 类工具的 deferred 行为（仅 `edit_file` / `rewrite_file`）。

## 4. 方案概览

```
┌─────────────────────────────────────────────────────────────┐
│  Agent Loop                                                  │
│  read/search … → edit_file (×N, 同文件可多次)                │
│       │                                                      │
│       ├─ agentDeferredEditReview=true                       │
│       │     → 跳过 tool_request 暂停（等同 auto-approve）    │
│       │     → apply diff，不 auto-accept                     │
│       │     → 记录 uri 到 thread.agentDeferredEditUris       │
│       └─ 继续 loop，不等待用户                               │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼ LLM 无 tool call
┌─────────────────────────────────────────────────────────────┐
│  awaiting_batch_review                                       │
│  侧边栏底部：「N 个文件待确认」 Accept All / Reject All      │
│  命令栏同步显示变更文件列表                                   │
└─────────────────────────────────────────────────────────────┘
```

## 5. 配置项

新增全局设置（`GlobalSettings`）：

| 字段 | 类型 | 默认 | 说明 |
|------|------|------|------|
| `agentDeferredEditReview` | `boolean` | `true` | Agent 模式下 edit 工具延迟批量 review |

生效条件：`chatMode === 'agent'` **且** `agentDeferredEditReview === true` **且** 工具为 `edit_file` / `rewrite_file`。

## 6. Thread 状态

`ThreadType.state` 新增（非持久化语义，随 thread 序列化无害）：

```typescript
agentDeferredEditUris: string[]  // fsPath，本 agent 轮次内 deferred edit 触及的文件
```

## 7. Stream 状态

`IsRunningType` 新增：

```typescript
| 'awaiting_batch_review'  // Agent 已结束，仍有 pending diff 待用户确认
```

与 `awaiting_user`（工具权限）区分：前者是 **diff 批量确认**，后者是 **执行前授权**。

## 8. 核心逻辑（chatThreadService）

### 8.1 `_runToolCall` — 自动批准 edit 工具

```typescript
const deferEditReview =
  chatMode === 'agent'
  && globalSettings.agentDeferredEditReview
  && (toolName === 'edit_file' || toolName === 'rewrite_file')

const autoApprove =
  globalSettings.autoApprove[approvalType]
  || deferEditReview
```

仍写入 `tool_request` 消息（UI 可显示 edit 预览），但不 `return { awaitingUserApproval: true }`。

edit 成功后 append `uri.fsPath` 到 `agentDeferredEditUris`（去重）。

### 8.2 `_runChatAgent` 结束 — 进入批量 review

Agent loop 退出且 `isRunningWhenEnd !== 'awaiting_user'` 时：

```typescript
if (_hasPendingDeferredEdits(threadId)) {
  isRunningWhenEnd = 'awaiting_batch_review'
}
```

`_hasPendingDeferredEdits`：遍历 `agentDeferredEditUris`，检查 `editCodeService.diffAreasOfURI` 是否有未 accept 的 DiffZone。

### 8.3 批量 accept / reject

```typescript
acceptAllDeferredEdits(threadId)
rejectAllDeferredEdits(threadId)
dismissDeferredEditReview(threadId)  // 仅清除 awaiting 状态，不动 diff
```

新用户消息或 `abortRunning` 时调用 `dismissDeferredEditReview` 并清空 `agentDeferredEditUris`。

## 9. 同一文件多次 edit

依赖现有 `instantlyApplySearchReplaceBlocks` / `instantlyRewriteFile`：

- `startBehavior: 'keep-conflicts'`：第二次 edit 前 reject 旧 diff 装饰但 **保留已写入的 model 内容**，新 DiffZone 的 `originalCode` 仍为会话首次 edit 前快照。
- LLM 对 `read_file` 读到的是 **当前累积内容**，search/replace 可基于最新文本匹配。

无需额外 staging 文件。

## 10. UI 变更

### 10.1 `EditToolAcceptRejectButtonsHTML`

当 `agentDeferredEditReview && chatMode === 'agent'` 且 `isRunning ∈ { LLM, tool, awaiting_batch_review }` 时 **不渲染** 单条 Keep/Remove。

Agent 完全 idle 且用户关闭 deferred 或手动 review 后，恢复单条按钮（与命令栏二选一，不重复打扰）。

### 10.2 侧边栏 `AgentBatchReviewBar`

`isRunning === 'awaiting_batch_review'` 时在输入框上方展示：

- 文案：`{n} 个文件有待确认的修改`
- 按钮：Accept All / Reject All（调用 chatThreadService）
- 可选：Dismiss（保留 diff，清除 awaiting 状态）

### 10.3 线程状态指示

`awaiting_batch_review` → 黄色「待确认修改」，与 `awaiting_user`「Needs Approval」并列。

### 10.4 设置页

Chat 区域增加开关：**Agent 延迟审核文件修改**（`agentDeferredEditReview`）。

## 11. 与现有开关关系

| 设置 | deferred 模式下的行为 |
|------|----------------------|
| `autoApprove.edits = false` | 被 deferred 覆盖（agent edit 不暂停） |
| `autoAcceptLLMChanges = true` | **优先**：每次 edit 仍立即 accept（deferred 无 pending diff） |
| `autoAcceptLLMChanges = false` | 默认路径：累积 diff，结束批量 review |

## 12. 测试要点

1. Agent 写 md：连续 3 次 `edit_file` 同一文件，中间无 Approve 弹窗，loop 不中断。
2. Agent 结束后出现批量 review 条，Accept All 清除 diff，Reject All 回滚到首次 checkpoint 前状态。
3. `agentDeferredEditReview = false` 时行为与改前一致。
4. `normal` 模式 edit 仍走原审核流程。
5. 新用户消息可继续对话，awaiting 状态清除。
